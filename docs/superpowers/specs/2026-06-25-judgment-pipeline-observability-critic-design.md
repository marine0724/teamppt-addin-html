# 판단 파이프라인 — 관찰가능성 + 깐깐한 검수자 (Route B 2단계 토대) — 설계

날짜: 2026-06-25
관련: [에셋 조합 추천](2026-06-24-asset-combination-recommendation-design.md)(직전 — 추천 단계), [vibe-designing & A-first](2026-06-22-vibe-designing-a-first-design.md)

## 한 줄 요약

추천·배치 파이프라인을 **역할이 명확히 분리된 워크플로**로 재구성하고, 각 단계의 **판단 근거(reasoning)를 전부 가시화**한다. 마지막에 **실무 10년차 PPT 디자이너 수준의 깐깐한 검수자**가 실제 배치된 슬라이드 렌더를 보고 품질을 채점하며, 결과가 나쁠 때 **기능 / 데이터스키마 / 에셋** 중 무엇이 병목인지 진단한다.

## 배경 — 왜 이걸 하나

직전 스펙([에셋 조합 추천](2026-06-24-asset-combination-recommendation-design.md))으로 추천→배치까지 동작은 됐다. 그러나 실사용에서 두 가지 한계가 드러났다:

1. **구성기가 장님이다.** `CombinationRecommender`가 LLM에게 보내는 건 전부 텍스트(`file`/`name`/`capacity`/`useWhen`)뿐 — **에셋 썸네일(실제 생김새)을 한 번도 보지 않는다.** 따라서 "어떤 조합이 *예쁘게* 표현될지"라는 판단이 원천적으로 불가능하다.
2. **판단 근거가 안 보인다.** `thinkingBudget=0` + structured output이라 LLM이 *왜* 그렇게 골랐는지가 사라진다. (예: component가 "미충족"으로 빠졌을 때, 그게 ③ 구성기의 판단인지, ② 검색이 엉뚱한 후보를 물어온 건지, 애초에 풀에 없어서인지 구별 불가.)

## 목표 — "asset-bound 임계점"에 도달하기

사용자 확정 목표: **지금 가진 평범한 에셋으로도 어느 정도 굴러가고, 이제부터 결과가 나빠지는 유일한 이유가 "로직이 모자라서"가 아니라 "에셋이 아직 안 예뻐서"가 되는 지점**에 도달하는 것. 그게 0→1.

이 목표가 **완료 조건(Definition of Done)**을 정의한다:

> 모든 단계 신호(① 이해 정확도, ② 검색 유사도, ③ 구성 판단, 스키마 추출)가 *건강*한데, 오직 "잘 맞는·예쁜 에셋" 하나만 부족한 상태가 trace에서 재현되면 → 이 스펙의 개발은 끝. 그때부턴 에셋만 부으면 점점 좋아진다.

즉 trace는 단순 디버깅 도구가 아니라 **"개발을 언제 멈춰도 되는가"를 알려주는 계기판**이다.

## 핵심 불변 원칙 (유지)

- **사실 = COM / 벡터, 판단 = LLM.** 텍스트·개수·도형은 COM 원문, 검색 유사도는 벡터. 역할·조합·품질 판정만 LLM.
- **LLM은 텍스트 내용을 생성·수정하지 않는다.** 검수자도 "평가"만 하고 슬라이드 내용을 고치지 않는다.
- API 키를 문서·커밋에 평문으로 넣지 않는다.
- 좌표 변환(CoordinateConverter) 폴백 추가 금지.

## 에이전트 vs 워크플로 — 명시적 결정

이 시스템은 **워크플로**다(에이전트 아님). 단계 순서와 횟수가 고정이고, LLM이 "다음에 뭘 할지"를 스스로 정하지 않는다. 이유: "이쁜 레이아웃 고르기"는 매번 같은 순서로 풀려야 하는 결정적 문제이며, 열린 에이전트 루프는 비용·지연·디버깅·503 노출 면적을 모두 키운다. (재검색 루프 같은 agency는 단순 워크플로가 부족하다고 *판명될 때* 추가 — 지금은 YAGNI.)

## 설계

### 1. 파이프라인 모양 (5역할 → 실질 LLM 호출 2~3개)

```
① 이해 (멀티모달, LLM)    초안 렌더 + 재료  →  의도 · 필요조합 · 재료수 · reasoning
   [기존 DraftUnderstanding 확장: reasoning 필드 추가]

② 검색 (벡터, 비-LLM)     kind별 match_assets  →  후보풀(썸네일 경로 포함) · 유사도
   [기존 CombinationCandidateProvider — 이미 유사도 로깅]

③ 구성 (멀티모달, LLM) ★   초안 렌더 + 후보 썸네일들을 '눈으로' 보고
   [핵심 업그레이드]        →  조합 선택 · 슬롯별 fitNote · confidence · 미충족 · reasoning
                           철학: 완벽 아니어도 '있는 것 중 최선' 배치, 낮으면 confidence로 정직 표시

④ 배치 (COM, 비-LLM)      선택 조합  →  새 슬라이드에 shapes 합체  →  렌더 캡처
   [기존 PlaceOnNewSlide + SlideCaptureService 재사용]

⑤ 검수 (멀티모달, LLM)     배치된 슬라이드 렌더 1장 + 루브릭(텍스트)
   [신규, 온디맨드]         →  점수 · 평 · 병목진단(기능/스키마/에셋) · 제안 · reasoning
```

- ②④는 사실 영역(벡터·COM)이라 불변원칙에 부합.
- ③의 **핵심 변화 = 멀티모달**: 후보 썸네일을 실제로 본다. 더 예쁜 에셋이 들어올수록 자동으로 더 나은 조합 선택 = asset-bound.
- ⑤는 **온디맨드**(매번 X) — 비용 통제(아래 §4).

### 2. 관찰가능성 — RecommendationTrace

모든 LLM 단계가 **자기 판단 근거를 `reasoning` 필드로 결과에 같이 뱉도록 응답 스키마에 강제**한다. 오케스트레이터가 단계별 결과를 하나의 `RecommendationTrace` 객체로 모은다.

```
RecommendationTrace
 ├─ Understand  : { intent, neededCombination, counts, reasoning }
 ├─ Retrieve    : kind별 [ { name, similarity } ... ]   (LLM 아님 — 사실 기록)
 ├─ Compose     : { picks[ {kind, name, confidence, fitNote} ], unmet[], reasoning }
 └─ Critique?   : { score, dimensionScores{}, verdict, bottleneck, suggestion, reasoning }  (검수 실행 시만)
```

표면(노출):
1. **UI 접이식 "🔍 판단 과정" 패널** — 추천 카드 아래. 각 단계 reasoning을 사람이 읽을 수 있게.
2. **`debug.log`** — 동일 내용을 `[Trace]` 태그로. (사후 분석용)

UI 표시 예:
```
🔍 판단 과정 (펼치기)
 ① 이해:  "표지가 아니라 본문. 비전 설명형. 좌측 텍스트 + 우측 시각 필요."
 ② 검색:  header 5(sim 0.70~0.72) / layout 5(0.70~0.76) / component 5(0.63~0.71)
 ③ 구성:  "layout '좌측번호+우측이미지' 선택(90%). component 후보 5개가 다 차트류라
           이 본문엔 과해서 미충족."
 ⑤ 검수:  78점 — "위계는 좋으나 여백 답답. 병목=에셋(차트 일색)."
```

### 3. 깐깐한 검수자 ⑤ — 전문 디자이너 페르소나

- **페르소나:** "실무 10년차 PPT 에이전시 시니어 디자이너. 깐깐하다."
- **입력:** 배치된 슬라이드의 **실제 렌더 이미지 1장** + 초안 렌더 + 루브릭(텍스트) + 단계 trace 요약
- **"실무급"의 정의 = 가중 루브릭(레퍼런스 이미지 아님):**

  | 차원 | 배점 | 본다 |
  |---|---|---|
  | 정렬·격자 | 20 | 요소가 보이지 않는 격자에 맞나 |
  | 여백·호흡 | 15 | 답답하지 않나, 여백이 의도적인가 |
  | 위계 | 20 | 제목>소제목>본문 시선 흐름이 명확한가 |
  | 색 대비·조화 | 15 | 배경/텍스트 대비, 팔레트 일관성 |
  | 타이포 | 10 | 폰트 위계·자간·줄간격 |
  | 초안 의도 부합 | 20 | 이 슬라이드가 의도(예: '비전 설명')를 실제로 잘 하나 |

  합계 100 · **실무급 = 80↑ · 평범 = 60~79 · 미달 = <60**. → "실무급"이 vibes가 아니라 **튜닝 가능한 숫자**.

- **출력(구조화):**
  - `score` (0~100) + `dimensionScores` (차원별)
  - `verdict` (한두 문장 총평)
  - `bottleneck` ∈ {기능, 데이터스키마, 에셋} + 이유
  - `suggestion` (다음에 뭘 고칠지 한 줄)
  - `reasoning`

- **병목 3분류를 자기-상대 기준으로 진단(추가 이미지 0장):** 검수자에게 "결과물이 *재료가 된 원본 에셋들*보다 못나 보이면 → 배치/기능 문제. 원본만큼은 나오는데 그래도 별로면 → 에셋 천장(=에셋 문제). 의도와 구성이 애초에 어긋났으면 → 스키마/이해 문제"라고 지시. 원본 에셋 품질은 ③에서 이미 썸네일로 봤으므로 별도 이미지 불필요.

### 4. 토큰 통제 (3겹)

1. **루브릭 = 텍스트, 렌더 = 결과물 1장만.** 비교용 레퍼런스 이미지 0장. (가장 큰 절감)
2. **검수자는 온디맨드.** "🔍 디자이너 검수 받기" 버튼 또는 ③ confidence가 임계 미만일 때만 발동. 일상 추천은 ③의 자가 confidence로 충분.
3. **루브릭·페르소나는 system prompt → prompt caching.** 반복 호출 비용 ≈ 0.

루브릭은 **살아있는 산출물**: v1은 손으로 박고(위 표), 추후 진짜 프로 덱 몇 개로 1회성 캘리브레이션해 정교화 후 저장. (지금은 v1으로 시작 — YAGNI.)

### 5. Graceful degrade (③의 철학)

평범한 풀에서 "완벽한 게 없다"고 판단될 때, **빈손이 아니라 "있는 것 중 최선 + 정직한 confidence"**로 채운다. 단 너무 낮으면(예: 30% 미만) 자동 제외 가능(선택). 사용자가 confidence를 보고 판단할 수 있게 항상 표시. → 평범한 에셋으로도 "어느 정도 구동"이 보장되어 임계점 측정이 가능해진다.

## 재사용 vs 신규

| 단계 | 현재 코드 | 이번 작업 |
|---|---|---|
| ① 이해 | `DraftUnderstandingService` (멀티모달 이미 함) | 응답 스키마에 `reasoning` 추가 |
| ② 검색 | `CombinationCandidateProvider`, `MatchQuery` (유사도 로깅) | trace로 유사도 수집 |
| ③ 구성 | `CombinationRecommender` (텍스트 전용) | **멀티모달화** — 썸네일 입력 + `reasoning` |
| ④ 배치 | `AssetPanel.PlaceOnNewSlide`, `SlideCaptureService` | 배치 후 렌더 캡처 연결 |
| ⑤ 검수 | 없음 | **신규** — `DesignCritiqueService` + 루브릭 + UI |
| Trace | 없음 | **신규** — `RecommendationTrace` 모델 + UI 패널 |

## 테스트 전략

- **순수 로직 TDD(비-UAC):** 파서들(reasoning 포함 응답, 검수자 응답 파싱), 루브릭 점수 합산, 병목 분류 파싱, Trace 직렬화. `RegisterForComInterop=false` + `vstest`로 UAC 없이.
- **수동 검증(UAC/COM):** PowerPoint에서 추천→배치→검수 전체 흐름, "🔍 판단 과정" 패널 표시, 검수 점수·병목 진단 출력.

## 비고 / 보류

- ③ 멀티모달에서 후보 썸네일을 몇 장까지 한 번에 보낼지(토큰) — 구현 시 측정해 상한 결정.
- 검수자 재검색/재배치 자동 루프 — 지금은 **진단·보고만**, 자동 수정 안 함(워크플로 유지). 필요 판명 시 별도 스펙.
- 루브릭 캘리브레이션(프로 덱 학습) — 별도 1회성 작업, 이번 범위 밖.
