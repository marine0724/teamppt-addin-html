## Extract (한눈에)

리디자인(한방 변환)이 제품의 정점이라는 결론은 **그대로 유지**한다. 그러나 **지금 바로 B(리디자인)로 달려가는 로드맵은 위험**하다 — A가 어설픈 상태에서 B를 얹으면 무너진다. 그래서 **"온전한 A"를 먼저 완성**한다. A만 써도 감동을 주는 수준이 되면, A·B·C가 전부 이상적으로 굴러간다. 리디자인은 늦지 않는다.

그리고 **A를 완성하는 순서는 "데이터 먼저"** 다. 에셋은 R&D가 따로 붓는 병렬 트랙이라 그 레일(인제스트→Supabase 벡터추천)을 먼저 깔아야 한다. 같은 7개라도 "LLM 인제스트→Supabase 벡터추천"이냐 "수작업 json 로드"냐는 **투자자가 보는 비전이 하늘과 땅**이다.

## 1. 제품 정체성 — 바이브 디자이닝 (나라)

TEAMPPT의 정체성은 **바이브 디자이닝(Vibe Designing)** 이다. 커서가 "바이브 코딩"이듯, 우리는 디자인을 그렇게 만든다.

핵심 약속: **전문 역량 → 노멀 역량이 되어도 *같은 결과물*이 나온다.** (커서·클로드 코드가 코딩에서 그랬듯.) 지금 단계(A)에선 "무엇을 고를지"에 약간의 전문 역량이 필요하지만, 점점 노멀한 사람도 같은 결과에 닿게 만든다.

**A만 써도 엄청난 결과·감동**이 나와야 한다 — 사용자가 커서를 처음 썼을 때 받은 그 감동. A가 모든 것의 토대다.

## 2. 계층 구조 (진행보드 5단)

```
나라(國)  바이브 디자이닝 ── 전문역량 → 노멀역량이 되어도 같은 결과물
  └ 대지   [A 에셋조립·커서식]   [B 리디자인·위임식]   [C 기획+에셋]
              ▲ 지금 여기                (다음)             (추후)
       └ 숲   온전한 A 로드맵 (데이터 → 공유 → 추천·진단 UX)
          └ 나무   현재 plan
             └ 잎   task
```

- **나라 = 바이브 디자이닝** (정체성/비전)
- **대지 = A / B / C 루트** (바이브 디자이닝이 *어느 단위까지* 해주느냐)
- **숲 = 로드맵 / 나무 = plan / 잎 = task** (기존 유지)
- 진행보드 최상단에 나라·대지 두 줄을 새로 얹는다. 숲 이하 골격은 그대로.

## 3. 대지 — A/B/C 루트 (비유로 정의)

| 루트 | 비유 | 사용자의 위치 | AI가 묻는 것 | 진단의 목적지 |
|------|------|---------------|--------------|----------------|
| **A 에셋 조립** | 커서로 페어코딩 (바이브 코딩) | 슬라이드에 *직접 손이 닿아 있음* | "어떤 **에셋**?" | **부분 보강** (조립 관점) |
| **B 리디자인** | 클로드 코드로 위임 (바이브 디자이닝 정점) | 한 발 물러나 *전두지휘* (할 필요가 없어 안 하는 것) | "어떤 **컨셉**?" | **전체 변환** (컨셉 관점) |
| **C 기획+에셋** | 백지에서 구성부터 | (추후) | "어떤 **구성**?" | 본격 진단 리포트 |

- **A와 B의 유일한 차이 = "전체를 한 번에 보고 한 번에 갈아치우는 실행엔진"의 유무.** 둘 다 같은 '읽기' 엔진(현재 슬라이드/전체 덱 공유)을 공유한다. B는 그 위에 한방 변환 실행엔진을 얹은 것뿐.
- B 진입 구상: 고객이 초안 PPT를 연 상태 → AI가 훑어 진단 → 대화로 "리디자인 할까요?" → OK → **별도 버튼**으로 B 진입. 끝나면 B→A로 에셋을 하나씩 갈아끼울 수도.
- A·B는 **별개 기능**(섞지 않음). 비포애프터는 B에서만.

## 4. 이번 결정 — A 우선 + 데이터 먼저

**(1) A 우선:** D(한방 실행엔진)는 B 루트로 분리하고 후순위로 미룬다. 지금은 A를 온전하게.

**(2) 데이터 먼저 (갈래2 채택):** 후보였던 "경험 먼저(로컬 7개 위에서 루프부터)" 대신 "데이터 토대 먼저"를 택한 이유 —
- 에셋은 사용자가 아니라 **R&D가 따로 붓는 병렬 트랙**. 그 레일을 먼저 깔아야 그들이 계속 부을 수 있다.
- 같은 7개라도 **"LLM 인제스트→Supabase 벡터추천" vs "수작업 json 로드" = 투자자가 보는 비전이 하늘과 땅.**
- 데이터가 가장 베이스. 환경 구축은 이미 거의 다 돼 있어(인제스트 로컬코어 ✅, 제작 규약 ✅, Gemini ✅, Supabase·인제스트 설계 ✅) 진입장벽이 낮다.

## 5. "온전한 A" 정의 (완성 기준)

A는 다음이 되면 완성이다 — **커서식 페어 디자이닝:**

1. PPT를 열고 우측 TEAMPPT 패널을 켠다. 상단에 **`[슬라이드 4 공유중]`** 표시(가벼운 지속 추적, 토큰 0).
2. 채팅: *"이 슬라이드에 장점 3개를 아이콘으로 나열하고 싶어"* → AI가 **현재 슬라이드를 온디맨드로 읽고** + **Supabase 벡터검색으로 적합 에셋 추천**(2~3개) → 카드 제시 → 클릭/드래그 삽입.
3. 채팅/버튼: *"전체 덱 봐줘"* → **전체 덱 온디맨드 공유** → **일관성 진단**(헤더 일관성 · 컨셉(색) 일관성 · 전반 구성) + 에셋 추천.
4. **스타일 가이드**: 색/폰트/팔레트 추천·적용 (기존 StylePanel 확장).
5. **데이터 토대**: 에셋이 로컬 json이 아니라 **Supabase 벡터 DB에서 검색**됨. R&D 인제스트로 에셋이 계속 축적됨.

**A의 진단 범위(중요·좁게):** 현재 슬라이드 = 구체적 에셋 추천 + 가벼운 코멘트. 전체 덱 = *일관성*에 한정(헤더·색 컨셉·구성). **본격 진단 리포트(텍스트 과다/정렬/빈 슬라이드 등 넓은 진단)는 C로 미룬다.**

## 6. 온전한 A 로드맵 (숲)

| 순서 | 단계 | 무엇을 | 기존 Phase 흡수 |
|------|------|--------|------------------|
| **A-1** | **데이터 토대 (해자)** — *다음 깃발* | 인제스트 LLM 이해 어댑터 → 임베딩 → **Supabase 업로드** + **LLM이 Supabase에서 불러와 추천**(벡터검색 읽기경로). R&D가 에셋을 계속 붓는 레일. | B 인제스트 나머지 + F 벡터 |
| **A-2** | **실시간 공유 엔진** | 슬라이드 리더(현재/전체 온디맨드) + `[슬라이드 N 공유중]` indicator | C 슬라이드 리더(확장) |
| **A-3** | **추천·진단 UX 완성** | AI 탭 = 현재슬라이드 에셋추천(2~3개) + 전체덱 일관성 진단 | 기존 AI 탭 확장 |
| (분리) | **D 한방 실행엔진** | 전체를 한 번에 변환 → **B 루트로 이동·후순위** | 지금 안 함 |

각 단계는 자체 spec→plan→구현 사이클을 갖는다. 이 문서는 **A-1을 첫 plan 대상**으로 넘긴다.

## 7. 실시간 공유 = 온디맨드 + indicator (아키텍처 결정)

지속 동기화가 이상적이나 토큰 비용이 비현실적 → **온디맨드 채택.** 크롬의 Gemini 사이드패널과 같은 결.
- **공유 indicator**: 현재 활성 슬라이드만 가볍게 지속 추적 → `[슬라이드 N 공유중]` 표시. 토큰 0(번호/이름만). → 사용자는 *계속 함께하는 것처럼* 느낌.
- **실제 읽기·LLM 전송**: 온디맨드 — 채팅 칠 때 / 버튼 누를 때만.

## 8. 유지 / 수정 / 폐기

- **5대 결정 → 전부 유지** (두 토큰 예산 / 역할 색·폰트 / 슬롯 / 컨셉 레이어 / 비파괴). 이번 재정립이 오히려 이 결정들을 더 잘 살린다.
- **세 루트 → 유지 + 격상** (진행보드 `대지` 최상위층으로). 비유로 정의 보강.
- **로드맵 A→E→C→B→D → 재배치(수정)**: C·B나머지·F를 "온전한 A" 한 깃발로 묶고, **D만 B 루트로 분리·후순위.**
- **폐기 → 없음.** (캔버스 웹UI는 기존대로 보류 유지.)

## 9. 발표(피치) 관점 — 무엇을 기록할지

이 프로젝트는 곧 **대표·투자자 발표**에 쓰인다. 목적은 기술 자랑이 아니라, 비전문가가 **가치·낮은 위험·큰 성장 가능성**을 느끼고 감동받게 하는 것. ([[feedback-presentation]])
- 빌드하면서 **① 완성된 기능(시스템이 어떻게 도는지 도식화 가능하게) + ② 버그 방지용으로 설계된 기능**을 비전문가 언어로 [`docs/PITCH.md`](../../PITCH.md)에 누적 기록한다.
- ③앞으로 진행될 기능 ④그에 따른 UX ⑤정체성은 이미 본 문서·메모리에 박제 → 발표 시 그대로 인용.
