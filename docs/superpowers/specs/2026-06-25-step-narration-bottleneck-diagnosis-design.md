# 단계별 속마음 독백 + 병목 진단 정밀화 — 설계

날짜: 2026-06-25
관련: [판단 파이프라인(관찰가능성+검수자)](2026-06-25-judgment-pipeline-observability-critic-design.md)(직전 — 이 설계가 그 위에 얹힘)

## 한 줄 요약

각 파이프라인 단계의 `reasoning`을 **한국어 존대말 속마음 독백**으로 만들어 추천이 진행되는 동안 **실시간 AI 버블**로 흘리고, 동시에 trace를 **"지금 뭐가 병목인지" 정확히 가려주는 진단 계기판**으로 정밀화한다.

## 배경 — 왜 이걸 하나

직전 스펙으로 Trace·reasoning·검수자가 코드로 구현됐다(Task 1~10). PPT 검증에서 동작은 확인됐으나 두 가지가 드러났다:

1. **reasoning이 영어로/대충 나온다.** 프롬프트에 언어 지정이 없고(→ 영어), 모든 LLM 호출이 `thinkingBudget=0`이라(→ 즉답) "사람이 고민한 속마음"이 안 나온다.
2. **진단이 "구체적으로 뭐가 문제인지"까지 못 짚는다.** 사용자의 진짜 목적은 *개발 단계에서 병목을 콕 집어 없애는 것* — 데이터를 잘못 뽑았나 / 에셋이 부족(양)한가 / 에셋이 안 이쁜가(질) / 기능(배치·조립) 문제인가. **기능의 문제가 없어질 때까지** 가는 게 목표.

## 목표

추천 한 번을 돌리면, 단계별 독백 흐름만 읽고도 **위 4가지 병목 중 무엇인지**가 바로 읽혀야 한다. trace = "개발을 언제 멈춰도 되는가(=기능 문제가 사라졌는가)"의 계기판.

## 핵심 통찰 — 병목은 한 단계가 아니라 여러 신호의 종합

검수자(⑤)는 결과 렌더 1장만 보므로 "에셋이 풀에 부족했는지(양)"는 원천적으로 알 수 없다. 그건 ② 검색 유사도만 아는 사실이다. 따라서 진단은 **여러 단계 신호의 종합**이어야 정직하다:

| 무엇이 문제? | 어느 신호가 말해주나 |
|---|---|
| 데이터를 잘못 뽑음 (추출) | ① 이해의 자기진단 + ⑤ "의도 어긋남" |
| 에셋 부족 (양) | ② 검색 유사도가 낮음 |
| 에셋 안 이쁨 (질) | ⑤ "원본 에셋만큼은 나오는데도 별로" |
| 기능 (배치·조립) | ⑤ "결과가 원본 에셋보다 못나 보임" |

## 핵심 불변 원칙 (유지)

- **사실 = COM / 벡터, 판단 = LLM.** 유사도는 벡터 사실, 독백·채점은 LLM.
- **LLM은 텍스트 내용을 생성·수정하지 않는다.** 독백·검수는 평가·설명만.
- API 키를 문서·커밋에 평문으로 넣지 않는다.
- 자동 재검색/재배치 루프 추가 금지 — trace는 진단·보고만.

## 설계

### 1. 프롬프트 재설계 — 한국어 존대말 + 약점 솔직

세 스키마(`DraftUnderstandingSchema`, `CombinationRecommenderSchema`, `DesignCritiqueSchema`)의 `reasoning` 필드 지시를 교체:

- **한국어 존대말 명시** (현재 영어로 나오는 근본 원인 = 언어 지정 부재).
- **담백한 전문가 톤**, 1~2문장.
- **약점·아쉬운 점을 솔직히** 드러내라고 지시. ("문제없다"는 식의 무난한 답 금지. 어디가 애매하고 어디가 부족한지 짚게.)
- 각 단계 성격에 맞게:
  - ① 이해: 이 슬라이드를 어떻게 읽었고, **추출(counts/materials)에서 애매했던 부분**은 무엇인지.
  - ③ 구성: 왜 이 조합을 골랐고, **미충족이 났다면 풀에 없어서인지·다 안 맞아서인지**.
  - ⑤ 검수: 루브릭 채점 근거 + **병목이 무엇인지(아래 §4)**.

### 2. thinkingBudget 파라미터화 + 상향

`GeminiAiService.GenerateJsonAsync`가 하드코딩한 `thinkingBudget=0`을 **인자로 분리**(기본값 0 유지 → 기존 호출부 무영향). 단계별 지정:

- ① 이해 · ③ 구성: **768** (매번 호출 — 보수적).
- ⑤ 검수: **2048** (온디맨드 — 깐깐하게 보려면 실제로 더 봐야 함).

수치는 PPT 검증하며 조정. (향후 Claude로 엔진 교체 시 인터페이스 동일 — 지금 상향이 그대로 유효.)

### 3. ② 검색 유사도를 trace에 실측 노출 ⭐신규

`MatchQuery.ParseResults`가 이미 읽고서 **로그만 찍고 버리는** similarity(현재 28행)를 살린다:

- 각 `HeaderAsset.Extra["similarity"]`에 값을 싣는다.
- `CombinationCandidateProvider.LastRetrieveLines`를 개수만 → **개수 + 유사도 범위**로: 예 `"header 5개 (유사도 0.71~0.62)"`.
- 이것이 **에셋 양 부족을 가리는 직접 신호** — 후보가 5개여도 유사도가 다 0.5대면 "맞는 에셋이 풀에 없다".

### 4. ⑤ 검수자에게 trace 요약 입력 + 병목 4분류 ⭐신규

검수자 입력에 결과 렌더 + 초안 렌더뿐 아니라 **① 이해 요약 + ② 검색 유사도 요약**을 텍스트로 추가(토큰 거의 0). 그러면 검수자가 양/질/추출/기능을 **종합 판정**할 수 있다.

`DesignCritique.Bottleneck` enum을 사용자 4분류로 교체:

- 기존 `{기능, 데이터스키마, 에셋}` → **`{기능, 데이터추출, 에셋부족, 에셋품질}`**
- 검수 프롬프트의 병목 진단 지시도 4분류로 재작성:
  - **기능**: 결과가 원본 에셋보다 못나 보임(배치·조립 문제).
  - **에셋품질**: 원본 에셋만큼은 나오는데도 별로(에셋 천장 — 질).
  - **에셋부족**: 검색 유사도가 낮아 애초에 맞는 후보가 없었음(양). ← 입력으로 받은 유사도 요약 근거.
  - **데이터추출**: 의도·구성이 애초에 어긋남(이해·인제스트 문제).

`DesignCritiqueSchema.BuildResponseSchema`의 enum, `RecommendationTrace.ToReadableLines`의 병목 표시도 4분류 반영.

### 5. 실시간 버블 독백 배선

`RecommendationService.RunAsync`에 **독백 콜백** `Action<string> narrate`를 추가(기존 `progress`와 별개). 각 단계 완료 직후 호출:

- ① 이해 끝 → `narrate(u.Reasoning)`
- ② 검색 끝 → `narrate("후보를 추렸어요 — " + LastRetrieveLines 요약)` (사실 보고, LLM 아님)
- ③ 구성 끝 → `narrate(rec.Reasoning)`
- ⑤ 검수(버튼)는 이미 결과를 버블로 출력 중 → reasoning·병목까지 덧붙임(기존 `RunCritiqueAsync`).

`AssetPanel`은 `narrate`를 `AddAiBubble`에 연결. 기존 기계적 progress("초안 읽는 중…")는 **독백으로 대체**(단계 시작 시 "…중" 최소화, 단계 끝 독백이 주역).

### 6. 접이식 패널 유지

기존 `BuildTracePanel`/`ToReadableLines` 그대로. 같은 reasoning·유사도·병목이 패널에 요약으로 남아 나중에 다시 펼쳐봄.

## 재사용 vs 신규

| 항목 | 현재 | 이번 작업 |
|---|---|---|
| reasoning 필드 | 3개 스키마에 존재(영어/빈약) | 프롬프트 한국어·솔직 재작성 |
| thinkingBudget | `GenerateJsonAsync`에 0 하드코딩 | 인자로 분리 + 단계별 상향 |
| 검색 유사도 | `ParseResults`가 읽고 버림 | `Extra["similarity"]`로 살려 trace 노출 |
| 검수자 입력 | 렌더 2장만 | + 이해 요약 · 유사도 요약(텍스트) |
| 병목 분류 | `{기능,데이터스키마,에셋}` 3 | `{기능,데이터추출,에셋부족,에셋품질}` 4 |
| 버블 독백 | 기계적 progress만 | 단계별 reasoning 실시간 버블 |

## 테스트 전략

- **순수 로직 TDD(비-UAC):** `ParseResults`가 similarity를 `Extra`에 싣는지, `DesignCritiqueParser`가 4분류 enum을 파싱하는지, `ToReadableLines`가 유사도·4분류 병목을 렌더하는지.
- **수동 검증(UAC/COM):** PowerPoint 추천 시 ① 한국어 독백 버블이 단계별로 흐르는지, ② 유사도가 버블·패널에 보이는지, ③ 검수 병목이 4분류로 찍히는지, ④ `debug.log`의 reasoning이 영어가 아니라 한국어이고 약점을 짚는지.

## 비고 / 보류

- 엔진 Claude 교체는 별도 작업(이번 범위 밖). 인터페이스(독백 콜백·budget 인자)는 교체 후에도 동일.
- thinkingBudget 수치는 검증 후 조정(현 768/2048은 보수적 시작점).
- 검수자 자동 재배치 루프 — 여전히 진단·보고만(워크플로 유지).
